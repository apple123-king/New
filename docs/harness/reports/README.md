# Reports 目录说明

本目录用于存放验收、文档一致性、试玩观察和专项审查报告。

## 报告类型

| 报告 | 触发时机 | 负责人 |
|------|----------|--------|
| `iteration-N-review.md` | 当前迭代完成独立验收后 | `@reviewer` |
| `doc-consistency-*.md` | 文档清理或一致性审查后 | `@doc-cleaner` |
| `playtest-observation-*.md` | 本地试玩验证后 | `@planner` 或 `@reviewer` |

## 报告要求

- 必须写明检查对象、检查时间、结论、阻塞问题和非阻塞风险。
- 必须能回链到需求文档、任务合同或执行计划中的具体验收标准。
- 术语使用统一口径：
  - 已实现：实现已存在，但不代表独立验收已通过
  - 已验收：已经形成通过的正式报告
  - 已完成：通常只在进度与报告都已同步后使用
  - 当前代码现状：仓库中可观察到的事实状态

## 何时需要补录报告

以下情况应补录正式报告，而不是只更新进度：

1. 需要把某项实现正式认定为“已验收”。
2. 需要记录独立 reviewer 的通过或不通过结论。
3. 需要形成可回链的文档一致性审查结果。
4. 需要把“代码现状已存在”提升为“正式阶段依据”。

## 何时只更新进度即可

以下情况通常只更新 `docs/harness/progress.md`，不要求新增报告：

1. 只是同步当前正式阶段。
2. 只是补记当前代码现状。
3. 只是记录文档与代码之间的偏差。
4. reviewer 已有正式报告，但 `progress.md` 尚未同步。

## 与 error-log 的关系

如果报告发现架构、测试、文件分区或流程问题，需要同步追加到 `../error-log.md`。
