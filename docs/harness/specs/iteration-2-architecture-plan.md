# 迭代 2 架构规划

## 目标

在不直接实现完整联网版的前提下，为《双人彩弹躲藏对抗》建立更适合未来联网和真实信息保密的结构边界，同时补齐比迭代 1 更稳定的场景级自动化验证。

## 技术基线

- Unity: 2022.3.62f2c1
- UI: UGUI + TextMeshPro
- 测试: Unity Test Runner EditMode + PlayMode
- 代码风格: 纯 C# 规则核心 + 薄 MonoBehaviour 桥接 + 角色视图模型
- 当前实现状态: 迭代 1 已归档，迭代 2 处于规划阶段

## 模块边界

| 模块 | 职责 | 迭代 2 目标 |
|------|------|-------------|
| Core Rules | 比赛规则、射击、墙体、姿态、回合循环 | 保持稳定，不轻易改写已验收规则 |
| Command Boundary | 输入命令与权威调用之间的适配层 | 让未来联网输入更容易接入 |
| Role Visibility | 按攻击方、躲避方、调试模式裁剪只读信息 | 建立真实信息保密边界 |
| Scene Integration | 运行时自举、相机、输入转发、HUD 装配 | 从“能跑”升级为“可验证、可裁剪信息” |
| PlayMode Validation | 场景级自动化验证 | 减少只靠 Unity Skills 临时取证 |

## 通信方式

1. 玩家输入先进入命令边界，而不是由场景脚本直接改规则内部状态。
2. 规则核心继续输出权威结果，但由角色视图层决定某个结果是否可被攻击方、躲避方或调试模式看到。
3. HUD、场景对象和相机逻辑优先读取角色视图模型，不直接读取所有内部控制器细节。
4. Scene Integration 负责装配依赖与运行时对象，不承担新的复杂业务规则。
5. PlayMode 测试优先从运行时入口验证“场景桥接 + 角色视图 + 基本稳定性”。

## 方案选择

### 采用: 在现有规则核心外补一层角色视图与命令边界

- 优点: 改动集中、复用迭代 1 成果、为联网版保留演进路径。
- 风险: 需要先统一共享合同和只读视图结构。
- 当前状态: 计划采用。

### 不采用: 直接在现有 HUD 上加 if/else 隐藏信息

- 问题: 表面上能遮 UI，但底层数据边界仍混乱，后续联网和 reviewer 验证都不稳定。

### 暂不采用: 本迭代直接接入完整 NGO/房间系统

- 问题: 会把“玩法验证后的边界整理”与“真实联网实现”混成一个高风险大步。

## 风险与约束

- `GameSliceContracts.cs` 可能继续承担共享边界演进，任何改动都要同步检查现有 EditMode 测试。
- `SceneIntegration` 与 `DodgerAndUI` 在本迭代中耦合较高，任务拆分时必须明确独占文件。
- PlayMode 测试引入后，测试链路会比迭代 1 更复杂，需要注意运行时间与稳定性。
- 不允许为了做信息保密而破坏迭代 1 已通过验收的比赛核心规则。
