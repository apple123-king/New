# 需求文档: 迭代 2 信息保密验证与联网前置架构

## 基本信息

- **迭代**: 迭代 2
- **优先级**: 高
- **复杂度**: 复杂
- **依赖**: Unity 2022.3.62f2c1、UGUI/TextMeshPro、Unity Test Runner、Unity Skills、迭代 1 已归档基线

## 功能描述

在迭代 1 已验证本地双人 3D 灰盒垂直切片玩法成立的前提下，迭代 2 的目标不是直接做完整联网版，而是先验证进入联网版前最关键的三个前置问题：

1. 真实非对称信息保密是否能在现有灰盒切片上成立。
2. 当前代码能否抽出清晰的“输入命令 -> 权威规则 -> 角色视图”边界。
3. 关键场景级行为能否从临时运行态取证，提升到更稳定的 PlayMode 自动化验证。

## 输入

- 攻击方输入命令：观察、开镜、开火
- 躲避方输入命令：移动、姿态切换
- CoreLoop、Shooter、WallDestruction、DodgerAndUI 既有规则输出
- SceneIntegration 的运行时桥接事件
- 角色身份与视图权限定义

## 输出

- 攻击方视角专属 HUD 与状态信息
- 躲避方视角专属 HUD 与状态信息
- 调试全信息模式与正式信息保密模式的切换策略
- 更清晰的命令入口、权威结算结果与角色只读视图边界
- 至少一条可复用的 PlayMode 场景级自动化验证链路

## 行为规则

1. 迭代 1 已通过验收的核心比赛规则继续保持不变，避免在同一迭代同时重写玩法规则和信息边界。
2. 默认正式验证口径下，攻击方与躲避方不应再共享完整调试信息。
3. 攻击方可以看到自己的剩余子弹、瞄准状态与射击反馈；躲避方不应直接看到攻击方剩余子弹数字。
4. 调试全信息模式可以保留，但必须显式标记为调试用途，不能与正式信息保密模式混淆。
5. 场景桥接层应把“玩家输入采集”与“权威规则调用”分开，避免未来联网改造时仍由 MonoBehaviour 直接操纵内部状态。
6. 规则核心输出应优先组织成角色可消费的只读快照或视图模型，而不是由 UI 或场景对象直接读取内部控制器细节。
7. 至少一项场景级关键行为应从人工运行态取证升级为 PlayMode 自动化验证，例如运行时自举、双相机创建、角色视图存在或基础输入链路。
8. 本迭代仍不实现完整好友房、断线重连或平台接入，只为联网版 PRD 前置降风险。

## 验收标准

1. [ ] 已建立攻击方、躲避方与调试模式的最小信息边界定义。
2. [ ] 运行时 HUD 与状态显示不再默认泄露双方完整信息。
3. [ ] 已抽出至少一层清晰的命令入口或权威适配边界，降低后续联网改造耦合。
4. [ ] 已补充至少一条 PlayMode 场景级自动化验证。
5. [ ] 不破坏迭代 1 已验收的核心规则与场景桥接基线。
6. [ ] Unity 编译检查通过，Console error 为 0。
7. [ ] reviewer 能基于需求、合同、场景验证和测试结果给出独立结论。

## 技术备注

- 优先复用迭代 1 中已经验收的纯 C# 规则核心，不轻易回退到“场景层直接包含业务规则”。
- 联网版前置目标是建立边界，不是一次性引入完整 NGO 或房间系统。
- PlayMode 自动化优先覆盖“场景桥接是否成立”，而不是立刻追求全量端到端对战自动化。
